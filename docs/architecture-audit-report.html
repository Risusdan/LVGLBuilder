LVGLBuilder 架構審計報告

執行摘要

本報告對 LVGLBuilder 專案進行架構層級的技術審計。該專案包含約 130 個原始碼檔案、約 14,000 行程式碼（不含 vendored 的 LVGL 與 FreeType 函式庫），目前零測試覆蓋率。審計共識別出 7 項架構問題（3 項高嚴重度、3 項中嚴重度、1 項低嚴重度）。核心結論是：在加入新功能之前，必須先進行架構重構；否則每次新增功能都會使技術債加倍累積，最終導致專案難以維護與擴展。

問題 1：上帝類別（God Class）高

問題描述

LVGLCore 是一個 818 行的單一類別，卻承擔了 8 項截然不同的職責：顯示驅動管理、圖片資源管理、字型管理、Widget 類型註冊、物件實例追蹤、顏色轉換、輸入處理、以及座標工具函式。用 C/C++ 的概念來說：想像一個單一的 .c 檔案，透過一個全域 struct 同時管理螢幕緩衝區、字型快取、圖片快取、Widget 工廠、以及物件註冊表。任何一個子系統的修改都可能影響其他子系統。

程式碼證據

LVGLCore.h:120-138 — 8 個不同的資料領域集中在同一個類別
// LVGLCore.h — 8 distinct data domains in one class
QHash<QString, LVGLImageData *> m_images;      // Image management
QHash<QString, const LVGLWidget *> m_widgets;   // Widget registry
QList<LVGLObject *> m_objects;                  // Object tracking
QList<LVGLFontData *> m_fonts;                  // Font management
lv_style_t m_screenStyle;                       // Style management
std::vector<lv_color_t> m_dispFrameBuf;         // Display buffer
lv_indev_data_t m_inputData;                    // Input handling
struct FT_LibraryRec_ *m_ft;                    // FreeType library

影響

  • 無法修改圖片管理而不冒著破壞字型管理或顯示渲染的風險
  • init() 函式（LVGLCore.cpp:34-130）必須一次初始化所有子系統 — 沒有辦法只啟動其中一個服務
  • 新進開發者必須理解 818 行程式碼後，才能安全地修改任何部分

建議解決方案

  • 方案 A：將每個職責提取為獨立類別（LVGLDisplay、LVGLImageManager、LVGLFontManager、LVGLWidgetRegistry、LVGLObjectManager）。LVGLCore 變成一個薄門面（facade），將請求委派給各服務。
  • 方案 B：至少將兩個最大的領域（圖片管理與字型管理）提取為獨立類別，可減少 LVGLCore 約 40% 的程式碼。

問題 2：全域單例耦合（Global Singleton Coupling）高

問題描述

整個應用程式依賴於一個單一全域變數 extern LVGLCore lvgl;（LVGLCore.h:142）。這等同於 C 語言中的一個巨大 extern struct — 任何包含該標頭檔的檔案都可以讀寫全域狀態。共有 24 個原始碼檔案直接呼叫 lvgl.someMethod()。

程式碼證據

LVGLCore.h:142 — 全域變數宣告
extern LVGLCore lvgl;
LVGLProject.cpp:98 — 專案儲存依賴全域狀態
for (LVGLObject *o : lvgl.allObjects()) {
    if (o->parent() == nullptr)
        widgetArr.append(o->toJson());
}
LVGLSimulator.cpp:320-329 — UI 渲染依賴全域狀態
QList<LVGLObject *> LVGLSimulator::objectsUnderCoords(
    QPoint pos, bool includeLocked) const {
  QList<LVGLObject *> ret;
  QRect screen = QRect(QPoint(0, 0), lvgl.size()).adjusted(-50, -50, 50, 50);
  for (LVGLObject *o : lvgl.allObjects()) {
    if (!includeLocked && o->isLocked()) continue;
    QRect geo = o->geometry();
    if (screen.contains(geo) && geo.contains(pos)) ret << o;
  }
  return ret;
}

影響

  • 無法測試 LVGLProject::save() 而不初始化整個 LVGL 顯示子系統、FreeType 和所有 Widget 類型
  • 24 個檔案中的任何一個 bug 都可能破壞全域狀態，導致不相關的檔案出錯
  • 不可能執行兩個獨立的實例（例如多文件編輯）

  • 方案 B：至少將 LVGLCore 作為明確參數傳遞給使用它的函式。

問題 3：零測試覆蓋、無法測試高

問題描述

零自動化測試。每次修改都只能透過 GUI 手動驗證。架構設計使得加入單元測試幾乎不可能：每個核心類別都依賴全域 lvgl 單例，而該單例需要完整的系統初始化才能使用。

程式碼證據

LVGLCore.cpp:34-63 — 測試任何功能都需要執行所有初始化
void LVGLCore::init(int width, int height) {
    lv_init();                          // 1. Initialize LVGL library
    m_dispFrameBuf.resize(n);           // 2. Allocate framebuffers
    lv_disp_drv_register(&m_dispDrv);   // 3. Register display driver
    lv_indev_drv_register(&indev_drv);  // 4. Register input driver
    m_default = lvgl.addImage(...);     // 5. Load default image
    m_fonts << new LVGLFontData(...);   // 6. Load all fonts
    addWidget(new LVGLArc);            // 7. Register 29 widget types
    addWidget(new LVGLBar);
    // ... 27 more registrations
}

影響

  • 14,000 行程式碼、零測試 = 每次修復 bug 都冒著引入回歸問題的風險
  • 手動測試每次變更需要數分鐘；自動化測試只需數秒
  • 在沒有安全網的情況下重構風險極高
  • 無法自動驗證 JSON 序列化、屬性邏輯或程式碼產生的正確性

建議解決方案

  • 方案 A：先分解 LVGLCore（問題 1），然後為每個提取出的服務撰寫獨立的單元測試。
  • 方案 B：加入整合測試，初始化完整系統後測試 JSON 往返（round-trip）和屬性 get/set 的正確性。

問題 4：UI 與業務邏輯混雜中

問題描述

LVGLSimulator::mousePressEvent() 在同一個事件處理器中包含了 Widget 選取、選單建構、物件刪除、重新排序、以及顏色對話框管理。用 C 的概念來說：一個 onMouseClick() 回呼函式同時處理碰撞偵測、選單顯示、物件刪除和設定。

程式碼證據

LVGLSimulator.cpp:139-194
void LVGLSimulator::mousePressEvent(QMouseEvent *event) {
    if (event->button() == Qt::RightButton) {
        auto obj = selectObject(...);        // Hit testing
        QMenu menu(obj->name(), this);       // UI: build menu
        QAction *remove = menu.addAction("Remove");
        sel = menu.exec(...);                // UI: show menu
        if (sel == remove) {
            lvgl.removeObject(obj);          // Business logic: delete
        } else if (sel == mfore) {
            lv_obj_move_foreground(obj->obj()); // Business logic: reorder
        }
    }
}

同樣地，MainWindow::loadProject()（MainWindow.cpp:263-280）混合了檔案 I/O、錯誤對話框、視窗標題更新、解析度變更和 UI 狀態管理。

影響

  • 無法在不啟動 Qt GUI 的情況下測試物件選取或刪除邏輯
  • 修改選單需要理解物件操作的程式碼
  • 新增操作意味著修改已經很複雜的函式

建議解決方案

  • 方案 A：提取 CanvasActions（建立/刪除/移動/重排序）和 SelectionManager。事件處理器變成輕量的分派器。
  • 方案 B：將每個事件處理器中的業務邏輯提取為獨立的私有方法。

問題 5：手動記憶體管理風險中

問題描述

整個程式碼中使用原始的 new/delete，多層指標鏈，沒有 RAII 或智慧指標。在 LVGLFontData::parse() 中，5 次連續的堆積配置 — 如果中途任何一次失敗，之前已配置的緩衝區就會洩漏。

程式碼證據

LVGLFontData.cpp:56-155 — 5 次連續原始配置，失敗時無清理機制
// 5 sequential raw allocations with no cleanup on failure:
lv_font_fmt_txt_glyph_dsc_t *glyph_dsc =
    new lv_font_fmt_txt_glyph_dsc_t[glyphs + 1];  // Alloc 1
uint8_t *gylph_bitmap_buf =
    new uint8_t[gylph_bitmap.size()];               // Alloc 2
lv_font_fmt_txt_cmap_t *cmap =
    new lv_font_fmt_txt_cmap_t[1];                   // Alloc 3
lv_font_fmt_txt_dsc_t *font_dsc =
    new lv_font_fmt_txt_dsc_t;                       // Alloc 4
lv_font_t *font = new lv_font_t;                    // Alloc 5

// Destructor must manually match all allocations:
~LVGLFontData() {
    delete [] fdsc->glyph_bitmap;
    delete [] fdsc->glyph_dsc;
    delete [] fdsc->cmaps;
    delete fdsc;
    delete m_font;
}

影響

  • 如果 FT_Load_Char() 在解析過程中失敗，已配置的緩衝區會洩漏
  • delete 與 delete[] 不匹配是未定義行為
  • 每位新進開發者都必須追蹤完整的配置鏈才能理解所有權關係

建議解決方案

  • 方案 A：使用 std::unique_ptr 搭配自訂刪除器。將相關的配置組合成 RAII 包裝結構。
  • 方案 B：至少使用 goto cleanup 模式（C 語言中常見的做法），確保每個離開路徑都能釋放所有配置。

問題 6：Widget 定義的重複程式碼中

問題描述

超過 20 個 Widget 檔案各自定義了結構幾乎相同的內嵌屬性類別。模式為：繼承 LVGLPropertyEnum，定義 name()、function()、get()、set() — 只有 LVGL API 名稱不同。

程式碼證據

widgets/LVGLButton.cpp:7-30
// widgets/LVGLButton.cpp
class LVGLPropertyButtonState : public LVGLPropertyEnum {
    QString name() const { return "State"; }
    QStringList function(LVGLObject *obj) const {
        return QStringList() << QString("lv_btn_set_state(%1, %2);")
            .arg(obj->codeName()).arg(m_values.at(get(obj)));
    }
    int get(LVGLObject *obj) const { return lv_btn_get_state(obj->obj()); }
    void set(LVGLObject *obj, int index) { lv_btn_set_state(obj->obj(), index & 0xff); }
};
widgets/LVGLLabel.cpp:7-28 — 相同模式，不同 API
// widgets/LVGLLabel.cpp — identical pattern, different API
class LVGLPropertyLabelAlign : public LVGLPropertyEnum {
    QString name() const { return "Align"; }
    QStringList function(LVGLObject *obj) const {
        return QStringList() << QString("lv_label_set_align(%1, %2);")
            .arg(obj->codeName()).arg(m_values.at(get(obj)));
    }
    int get(LVGLObject *obj) const { return lv_label_get_align(obj->obj()); }
    void set(LVGLObject *obj, int index) { lv_label_set_align(obj->obj(), index & 0xff); }
};

影響

  • 新增一個 Widget 需要複製約 80 行樣板程式碼
  • 模式中的 bug 必須在 20 個以上的檔案中逐一修復
  • 位於 Widget .cpp 檔案內的屬性類別無法被其他地方重用

建議解決方案

  • 方案 A：建立一個通用屬性模板，以 LVGL getter/setter 函式指標作為參數。
  • 方案 B：將所有屬性類別定義從 widgets/*.cpp 移到 properties/ 目錄，使其可被發現和重用。

問題 7：循環依賴與職責不清低

問題描述

LVGLObject 包含 LVGLCore.h 並直接呼叫 LVGL C 函式（lv_obj_del()），而 LVGLCore 儲存並管理 LVGLObject* 實例。這形成了雙向依賴。

程式碼證據

LVGLObject.cpp:3
#include "LVGLCore.h"
LVGLObject.cpp:43-50 — 繞過 LVGLCore 直接呼叫 LVGL
LVGLObject::~LVGLObject() {
    lv_obj_del(m_obj);
}
LVGLCore.h:125 — 儲存 LVGLObject 指標
QList<LVGLObject *> m_objects;

影響

  • 不清楚誰擁有 lv_obj_t 的生命週期管理權
  • 無法獨立編譯 LVGLObject 進行測試
  • 重構任一類別都需要同時理解兩者

建議解決方案

  • 方案 A：建立明確的所有權規則：ObjectManager 擁有並刪除物件。LVGLObject 永遠不直接呼叫 LVGL 生命週期函式。
  • 方案 B：使用前向宣告打破標頭檔的循環依賴。

額外效益

  • 新人上手：理解一個 200 行的服務 vs. 理解一個 818 行的上帝類別
  • 功能開發速度：變更局限於單一模組
  • Bug 隔離：字型管理的 bug 不會破壞圖片管理
  • 新功能整合：乾淨的介面有利於從 fork 移植功能

不重構的風險

每新增一個功能，耦合程度進一步加深
上帝類別朝 1,000 行以上成長
沒有測試的情況下，手動驗證時間隨功能數量線性增長
記憶體管理問題隨著更多 Widget 類型的加入而複合累積
團隊規模擴展變得不切實際
